## 1-2. D 세대 화면 원칙

- `product_generation`과 `model_family`은 MVP에서 백엔드가 `model_document_lineage.csv`를 `exact_sales_code` 기준으로 조회하여 파생한다.
- 향후 서비스 DB에서는 `ProductModelRegistry.product_generation`, `ProductModelRegistry.model_family` 정규 칼럼으로 관리한다.
- AI 상태 `GENERATING`과 혼동되지 않도록 화면·API 필드는 반드시 `product_generation`을 사용한다.
- 고객 화면에는 필요 시 `현재 지원 모델` 정도로 표시하고 세대 코드는 기술 상세에서만 제공한다.

- 검색 결과의 `product_model`, `product_code`, `scope_role`, `product_generation`, `model_family`이 현재 문의 제품 범위와 일치하는지 백엔드에서 검증한 뒤 사용한다.

# 2. 공식 근거 레지스트리 연결

## 2-1. 레지스트리·화면 DTO 필드

화면은 RAG 저장 파일이나 근거 레지스트리를 직접 조회하지 않는다. 백엔드가 검색 결과, 근거 레지스트리, 문서 메타데이터와 모델 계보 정보를 결합하여 `EvidenceCardDTO`를 반환한다.

```text
RAG 검색 결과
→ 근거 레지스트리·문서 메타데이터 조회
→ model_document_lineage.csv에서 제품 세대·모델 계열 파생
→ EvidenceCardDTO 생성
→ 고객·상담사·방문기사 화면 전달
```

### 원문·내부 경로 비노출

- `source_path`는 내부 파일 경로이므로 EvidenceCardDTO와 화면에서 제외한다.
- `ManualPage.text` 원문 전체는 고객·상담사·기사 화면에 전달하지 않는다.
- `retrieval_text`는 현재 RAG 물리 필수 필드로 요구하지 않는다.
- 내부 원문 검증이 필요한 경우 다음 조건으로 조회한다.

```text
DocumentChunk.document_id = ManualPage.document_id
AND
ManualPage.page IN DocumentChunk.page_refs
```

- 화면에는 `evidence_summary`, `section_title`, `page_refs`와 공식 출처 URL만 제공한다.
- 고객 화면에서는 `chunk_id`를 숨기고 상담사·기사 화면과 내부 로그에서만 사용할 수 있다.
- 프론트엔드는 근거 레지스트리 파일을 직접 조회하거나 공식 URL을 임의 조합하지 않는다.

## 2-3. 출처 버튼 규칙

### 고객 화면

- 최근 필터·카트리지 교체일

## 5-3. `CUST-03 AI 추가 질문`

AI는 고객이 이미 입력한 내용을 제외하고 `missing_fields`에 해당하는 질문만 표시한다.

| 대표 분류 | 보조 질문 예시 |
| --- | --- |
| 출수량 저하 | 단수, 호스, 필터, 다른 수전, 온수 사용, 수압 |
| 물맛·냄새 | 미사용 기간, 필터 시점, 공식 통수 후 지속 여부 |
| 제품 누수 | 물받이, 연결부 주변 물, 전원 분리, 원수 밸브 |
| 냉·온수 온도 이상 | 연속 출수, 잠금, 전원, 대기 시간, 표시 문구 |
| 기타 | 고객 원문과 미확인 표시를 저장하고 상담 연결 |

### AI 상태와 고객 표시

| 시스템 코드 | 고객 표시 |
| --- | --- |
| `IDLE` | 분석을 준비하고 있습니다. |
| `STRUCTURING` | 증상 정보를 분석하고 있습니다. |
| `CHECKING_MISSING_FIELDS` | 추가 확인 항목을 확인하고 있습니다. |
| `SAFETY_CHECK` | 안전 기준을 확인하고 있습니다. |
| `RETRIEVING` | 공식 문서를 검색하고 있습니다. |
| `RERANKING` | 관련 근거를 정리하고 있습니다. |
| `GENERATING` | 안내를 작성하고 있습니다. |
| `VALIDATING` | 결과와 안전성을 확인하고 있습니다. |
| `COMPLETED` | 분석이 완료되었습니다. |
| `FAILED` | 처리하지 못했습니다. 입력은 유지됩니다. |
| `CANCELLED` | 처리가 중단되었습니다. |

### 행동·이벤트

- `missing_fields`가 있으면 현재 AI 처리는 `COMPLETED(추가 정보 필요)`로 종료하고 문의 상태는 `QUESTIONNAIRE_IN_PROGRESS`로 유지한다.
- 추가 답변 제출 → `SUBMIT_ANSWERS` → 기존 입력을 유지한 채 `STRUCTURING`부터 다시 처리한다.
- 실패 시 `failed_stage`, `error_code`, `retry_count`를 기록하고 실패한 단계부터 재시도한다.
- 최대 재시도 초과 → `CONSULTATION_REQUIRED`

## 5-4. `CUST-04 공식 근거·안전 안내`

### 카드 순서

1. 현재 해야 할 행동
2. 위험도와 사용 제한
3. 공식 안전조치
4. 상담 조건
5. 공식 근거
6. 증상 요약
7. 금지 행동

### EvidenceCardDTO 사용

화면은 백엔드가 생성한 확장 `EvidenceCardDTO`를 데이터 소스로 사용한다.

프론트엔드는 사용자 역할별 표시 정책에 따라 필요한 필드만 노출하며,
내부 식별자와 검증용 필드는 일반 고객 화면에서 숨긴다.

```text
[공식 매뉴얼]
문서: WPU-JAC104D/JCC104D 사용설명서
버전: REV.00
페이지: 38
근거 요약: 제품 누수 발생 시 원수 밸브를 잠그고 전원을 분리한 뒤 상담
검증 상태: 공식 근거 확인 완료
[공식 출처 보기]
[문서 상세]
```

- `공식 출처 보기`는 `source_landing_url`을 사용한다.
- `설명서 PDF 열기`는 `source_direct_download_url`이 있을 때만 표시한다.
- 다운로드 URL이 없으면 문서명·버전·페이지·공식 랜딩 페이지만 표시한다.
- 고객 화면에는 `chunk_id`, 내부 경로와 원문 전체를 노출하지 않는다.
- `safe_actions`, `escalation_conditions`, `prohibited_actions`는 고객이 이해할 수 있는 문구로 표시한다.
- `data_classification`에 따라 공식·팀 설계·합성 시연 배지를 구분한다.

### 상태별 표시와 시스템 이벤트

CUST-04의 버튼은 문의 상태를 직접 변경하지 않는다. 화면 진입 전 백엔드가 AI 종료 사유를 다음 이벤트로 변환한다.

- 공식 근거·검증 통과 → `SAFE_GUIDANCE_READY` → `AI_GUIDANCE`
- 위험 규칙 일치 → `DANGER_DETECTED` → `CONSULTATION_REQUIRED`
- 공식 근거 없음 → `NO_EVIDENCE` → `CONSULTATION_REQUIRED`

`risk_level=danger` 또는 `requires_consultation=true`이면:

- `DANGER_DETECTED`
- 문의 상태를 `CONSULTATION_REQUIRED`로 전환
- 일반 자가조치 결과 화면 대신 `CUST-05 안전조치 확인 모드`로 이동
- 누수: 제품 전체 사용 중지, 원수 밸브·전원 분리
- 순간온수 경고: 음용 금지, 즉시 상담
- 해결됨·문의 종료 버튼을 제공하지 않음

## 5-5. `CUST-05 조치 결과·상담 요청`

### A. 일반·주의 문의 모드

입력:

- 조치 수행 여부
- 수행한 조치
- 결과
  - 해결됨
  - 일부 개선
  - 동일
  - 악화
  - 수행하지 않음
- 상담 요청

처리:

| 고객 입력 | 이벤트 | 다음 상태 |
| --- | --- | --- |
| 해결됨 | `CUSTOMER_REPORTED_SELF_RESOLVED` | 즉시 `RESOLVED` |
| 일부 개선·동일·악화·미수행 | `REQUEST_CONSULTATION` | `CONSULTATION_REQUIRED` |
| 상담 요청 | `REQUEST_CONSULTATION` | `CONSULTATION_REQUIRED` |

### B. 위험·상담 필수 모드

화면명은 `안전조치 확인·상담 요청`으로 표시한다.

입력:

- 원수 밸브 잠금 여부
- 전원 분리 여부
- 음용 중지 확인
- 상담 요청

제거:

- 일반 자가조치 해결됨
- 문의 종료
- 고객 자체 해결

안전조치 완료 여부와 상담 완료 여부를 별도 값으로 저장하며, 안전조치가 완료되어도 문의 상태는 `CONSULTATION_REQUIRED`다.

## 5-6. `CUST-06 문의 상세`

### 표시 정보

- 합성 데이터 배지
- 문의 ID
- 시나리오 ID: 시연 모드에서만 표시
- `product_code`, `manual_model`
- 문의 상태
- 현재 담당 주체
- 다음 처리 단계
- 고객 행동 필요 여부
- 방문 일정 상태
- 고객 방문 희망일
- 가상 방문 확정일
- 마지막 상태 변경 시각
- 공식 근거 카드
- 팀 설계 상태 이력
- 상담·방문 결과
- 현재 사용 안내 상태
- 고객 해결 피드백 제출 여부
- 담당자 최종 완료 여부

### 상태별 화면

#### `COMPLETION_PENDING`·고객 피드백 전

```text
처리 결과를 확인하고 해결 여부를 선택해주세요.
[해결됨] [해결되지 않음] [상담 재요청]
```

#### 고객이 해결됨 제출 후

```text
해결 피드백이 전달되었습니다.
담당자가 확인한 뒤 문의를 최종 완료합니다.
상태: 담당자 최종 확인 중
```

`COMPLETION_PENDING` 상태는 유지한다.

#### `RESOLVED`

- 처리 결과와 상태 이력만 조회
- 해결 여부 버튼 숨김

### 행동·이벤트

| 고객 선택 | 이벤트 | 처리 |
| --- | --- | --- |
| 해결됨 | `SUBMIT_RESOLUTION_FEEDBACK` | 피드백 저장, `COMPLETION_PENDING` 유지 |
| 해결되지 않음 | `CUSTOMER_REPORTED_UNRESOLVED` | `REOPENED` |
| 상담 재요청 | `REQUEST_CONSULTATION` | `CONSULTATION_REQUIRED` |

### 위험 표시 유지

- 누수는 담당자가 현재 사용 안내 상태를 갱신하기 전까지 사용 중지 표시를 유지한다.
- 순간온수 경고는 확인 완료 전까지 음용 금지를 유지한다.
- AI 결과만으로 안전 상태를 정상으로 변경하지 않는다.

## 6-2. `CONS-02 문의 상세·상담 기록`

### 우선 정보

1. 상담 필수 여부
2. 사용·음용 중지
3. 제품·구독 정보
4. 최근 관리일과 최근 필터·카트리지 교체일
5. 고객 원문과 문진·추가 답변
6. 고객이 수행한 조치와 결과
7. 공식 근거
8. AI 상담 요약 초안과 상담사 확정본
9. 상태 이력
10. 고객 해결 피드백

### 제품·관리 이력

| 화면 정보 | 시스템 필드 예시 |
| --- | --- |
| 제품 코드 | `product_code` |
| 매뉴얼 모델 | `manual_model` |
| 관리 유형 | `management_type` |
| 사용 시작일 | `service_start_date` |
| 최근 관리일 | `last_care_date` |
| 최근 필터·카트리지 교체일 | `last_filter_replacement_date` |
| 다음 케어 예정일 | `next_care_date` |
| 다음 케어 기준 | `next_care_basis` |

관리 이력은 고객이 다시 입력하게 하지 않고 제품·구독·케어 데이터에서 조회한다.

### EvidenceCardDTO 상세

- 문서·출처: `evidence_id`, `chunk_id`, `document_id`, `document_title`, `document_version`, `page_refs`, `section_title`, `source_type`, `provider`
- 근거·안전: `evidence_summary`, `risk_level`, `requires_consultation`, `safe_actions`, `escalation_conditions`, `prohibited_actions`, `verification_status`
- 제품 범위: `product_code`, `manual_model`, `product_generation`, `model_family`, `scope_role`
- 분류·링크: `data_classification`, `source_landing_url`, `source_direct_download_url`

`applicability`와 `allowed_use`는 백엔드 `RetrievalResult`에서 검증한다. 화면은 검증을 통과한 EvidenceCardDTO만 표시하며, 근거 레지스트리 미연결·S 세대·미검증 FAQ·제외 정책 데이터는 공식 안내 근거로 사용하지 않는다. `source_path`와 `ManualPage.text` 원문 전체는 전달하지 않는다.

### AI 상담 요약 검토·수정

AI는 고객 원문, 문진, 추가 답변, 자가조치 결과, 제품·관리 이력과 공식 근거를 바탕으로 상담 요약 초안을 생성한다.

```text
AI 상담 요약 초안
→ 담당 상담사 검토
→ 수정 또는 그대로 승인
→ 상담사 확정본 저장
→ 방문 필요 시 기사 사전 리포트의 입력으로 사용
```

| 화면 기능 | 처리 | 상태 전이 |
| --- | --- | --- |
| AI 요약 수정 | `UPDATE_CONSULTATION_SUMMARY` | 없음 |
| AI 요약 확정 | `CONFIRM_CONSULTATION_SUMMARY` | 없음 |
| 상담 시작 | `START_CONSULTATION` | `CONSULTATION_IN_PROGRESS` |

저장 필드 예시:

- `ai_consultation_summary`
- `confirmed_consultation_summary`
- `summary_confirmed_by`
- `summary_confirmed_at`

AI 초안과 상담사 확정본을 분리하여 저장하고, 기사 화면에는 상담사 확정본을 우선 제공한다.

| 행동 | 이벤트·기능 | 결과 |
| --- | --- | --- |
| AI 요약 수정 | `UPDATE_CONSULTATION_SUMMARY` | 상담사 수정본 저장, 상태 유지 |

## 8-3. AI 처리 상태

| 코드 | 화면 의미 |
| --- | --- |
| `IDLE` | 실행 전 |
| `STRUCTURING` | 증상 구조화 |
| `CHECKING_MISSING_FIELDS` | 누락 정보 확인 |
| `SAFETY_CHECK` | 위험 규칙 확인 |
| `RETRIEVING` | 공식 문서 검색 |
| `RERANKING` | 검색 결과 재정렬 |
| `GENERATING` | 안내·요약 생성 |
| `VALIDATING` | JSON·근거·안전 검증 |
| `COMPLETED` | 처리 완료 |
| `FAILED` | 처리 실패 |
| `CANCELLED` | 처리 중단 |

AI 상태는 문의 상태·방문 상태와 별도 필드로 저장한다.

### AI·RAG 내부 처리 순서

```text
STRUCTURING
→ CHECKING_MISSING_FIELDS
→ SAFETY_CHECK
→ RETRIEVING
→ RERANKING
→ 근거 레지스트리 검증
   - 제품 코드·매뉴얼 모델 일치
   - product_generation·model_family 일치
   - verification_status 확인
   - allowed_use·scope_role 확인
   - data_classification 확인
→ 검증 통과 근거만 GENERATING 전달
→ VALIDATING
→ COMPLETED
```

- 근거 레지스트리 검증은 별도 문의 상태가 아니라 `RERANKING` 후처리로 수행한다.
- 검증을 통과하지 못한 청크는 역할별 결과 생성에서 제외한다.
- 사용할 수 있는 공식 근거가 없으면 `NO_EVIDENCE` 이벤트로 `CONSULTATION_REQUIRED`에 전환한다.
- AI Orchestrator와 LLM은 근거와 권고만 반환하며 문의·방문 상태를 직접 변경하지 않는다.

## 8-4. 현재 사용 안내 상태

현재 사용 가능 범위는 제품 고장 상태가 아니라 고객에게 제공하는 안내이므로 `usage_status`를 사용하지 않는다.

| 코드 | 화면 표시 | 의미 |
| --- | --- | --- |
| `NORMAL` | 일반 사용 가능 | 별도의 사용 제한이 필요하지 않음 |
| `PARTIAL_STOP` | 일부 기능 사용 중지 | 특정 출수 또는 기능만 사용 중지 |
| `TOTAL_STOP` | 제품 전체 사용 중지 | 제품 사용 중지와 안전조치 필요 |
| `PENDING_CONSULTATION` | 판단 보류·상담 필요 | 공식 근거 부족 또는 추가 확인 필요 |

### 공통 시스템 필드

| 필드 | 화면 사용 |
| --- | --- |
| `usage_guidance_status` | 상태 배지와 사용 제한 영역 |
| `usage_guidance_message` | 고객에게 표시하는 구체적인 안내 |
| `restricted_functions` | 제한되는 출수·기능 |
| `guidance_basis` | 공식 근거·상담·방문 판단 근거 |
| `next_action` | 고객 또는 담당자의 다음 행동 |
| `current_assignee_type` | 현재 담당 주체 |
| `next_step` | 다음 처리 단계 |
| `customer_action_required` | 고객 행동 필요 여부 |
| `last_status_changed_at` | 마지막 상태 변경 시각 |
| `allowed_actions` | 현재 상태·역할에서 허용된 화면 행동·이벤트 목록 |

건강상 안전이나 수질을 보증하는 표현은 사용하지 않는다. 공식 근거가 부족하면 `PENDING_CONSULTATION`으로 표시하고 상담 연결을 제공한다.

### 버튼 표시·활성화 원칙

- 화면은 현재 상태만 보고 버튼 노출 여부를 자체 계산하지 않는다.
- Backend API가 State Machine과 권한 검증 결과를 반영한 `allowed_actions`를 반환한다.
- 버튼에 연결된 이벤트가 `allowed_actions`에 포함된 경우에만 버튼을 표시하거나 활성화한다.
- `allowed_actions`에 없는 이벤트는 버튼을 숨기거나 비활성화하고 서버에서도 재검증한다.

```text
예시
allowed_actions = [
  "SUBMIT_SYMPTOM",
  "REQUEST_CONSULTATION",
  "CANCEL_INQUIRY"
]
```

- 고객·상담사·기사 화면은 같은 문의라도 역할과 담당 여부에 따라 서로 다른 `allowed_actions`를 받을 수 있다.

### 인증·접근 제어 원칙

- 모든 고객·상담사·기사·운영 화면은 Backend API의 인증·권한 검증을 거친다.
- 세션이 없거나 만료되면 데이터 조회·상태 변경 이벤트를 실행하지 않는다.
- 본인 또는 담당자 권한이 없는 문의의 상세 데이터는 반환하지 않는다.
- 운영 담당자는 P1 조회·집계 범위만 허용하며 업무 상태를 직접 변경할 수 없다.
