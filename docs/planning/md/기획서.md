# 1. 프로젝트 주제

## 1-1. 프로젝트 개요

| 구분                   | 내용                                                                     |

| 연계 기업              | SK인텔릭스, SK인텔릭스서비스                                             |
| 핵심 사용자            | 정수기 구독 고객, 고객센터 상담사, 방문기사, 서비스 운영 담당자          |
| 기본 MVP 제품          | 상품 코드 WPUJAC104DWH / 매뉴얼 적용 모델 WPU-JAC104D·WPU-JCC104D REV.00 |

## 1-2. 주제 정의

> 고객이 정수기 증상을 한 번 입력하면 제품·관리 이력과 공식 문서 근거를 바탕으로 다음 행동을 안내하고, 필요한 정보를 상담사와 방문기사에게 연결하는 LLM·RAG 기반 업무지원 시스템이다.

## 1-3. 핵심 서비스 범위

- 제품·구독 정보와 케어 일정 관리
- 사전 문진, 자연어 증상 구조화와 위험도 판단
- 공식 문서 RAG 기반 고객 안내와 상담·방문 인계
- 상담·방문 결과, 케어 이력과 고객 후속 해결 여부 관리

### MVP 제외·제약

MVP에서는 실제 CRM 연동, 기사 자동 배정·예약, 외부 알림과 결제·해지·환불을 제외한다. 데이터는 가명·합성 정보만 사용하며, AI는 고장 확정 진단이나 제품 분해·부품 교체 등 위험한 직접 수리를 안내하지 않는다.

## 1-4. 대표 이용 시나리오

> **대표 이용 시나리오 이미지의 텍스트 변환**

> 1. 고객이 구독 중인 정수기를 선택하고 “정수기 물 나오는 양이 너무 약해졌어요.”라고 증상을 입력한다.
> 2. AI가 필요한 추가 정보를 질문하고, 공식 근거를 바탕으로 안전한 자가 확인 방법을 안내한다.
> 3. 고객이 필터 확인 등 안내된 조치의 수행 여부와 조치 후 증상 상태를 입력한다.
> 4. 상담사가 고객 정보, 제품 정보, AI 요약, 공식 근거와 고객 조치 결과가 포함된 문의 요약을 확인한다.
> 5. 상담사가 방문점검이 필요하다고 판단하면 문의를 방문점검 단계로 전환하고 가상 기사와 방문 일정을 등록한다.
> 6. 방문기사가 방문 전 증상, 고객 조치, 상담 결과, 우선 점검 항목과 공식 근거가 포함된 사전 점검 리포트를 확인한다.
> 7. 방문기사가 현장에서 점검한 결과, 수행 조치, 교체 여부와 처리 상태를 등록한다.
> 8. 방문 결과가 케어 이력에 반영되고, 고객은 처리 결과와 갱신된 다음 케어 일정을 확인한다.

# 2. 문제 정의

## 2-1. 핵심 문제

> **제품 정보, 관리 이력, 고객 증상과 조치 결과가 상담·방문 단계로 이어지지 않아 동일한 설명·검색·확인이 반복된다.**

| 사용자      | 현재 겪는 문제                                                                 | 업무·경험상 영향                              |
|-------------|--------------------------------------------------------------------------------|-----------------------------------------------|
| 고객        | 상담 때 말했던 내용을 기사가 방문할 때 다시 말해야 한다.                       | 문의 피로, 정보 누락, 다음 행동의 불명확성    |
| 상담사      | 흩어져 있는 매뉴얼을 찾아서 정리하느라 오랜 시간을 소모하게 된다.              | 상담 준비 지연, 반복 질문, 응대 편차 가능성   |
| 방문기사    | 상담사로부터 전달받은 내용이 충분하지 않아 원인을 정확히 모르고 방문하게 된다. | 현장 재확인, 준비 누락, 점검 시간 증가 가능성 |
| 운영 담당자 | 문의·상담·방문·후속 결과의 상태와 예외를 한 흐름으로 보기 어렵다.              | 지연·예외 탐지와 처리 결과 추적의 어려움      |

## 2-2. 현재 문제와 해결 방향

> **현재 문제와 해결 방향 이미지의 텍스트 변환**

> **Before**
> - 고객이 상담사에게 증상을 설명한다.
> - 상담사는 관련 매뉴얼을 직접 검색하고 내용을 다시 정리한다.
> - 방문기사는 충분한 인계 정보가 없어 현장에서 고객에게 같은 내용을 다시 묻는다.
> - 단계 사이에서 정보가 끊겨 반복 설명, 시간 낭비, 지연과 재확인이 발생한다.

> **After**
> - 고객이 제품과 증상을 한 번 입력한다.
> - AI가 공식 근거를 검색해 고객 안내와 상담 요약을 생성한다.
> - 상담 결과가 방문기사용 사전 점검 리포트로 이어진다.
> - 방문 결과가 원 문의와 케어 이력에 반영되고 다음 일정까지 갱신된다.

> 즉, 문제는 정보가 없는 것이 아니라 **단계가 바뀔 때 정보가 끊기는 것**이며, 본 프로젝트는 동일한 `inquiry_id`로 이를 연결한다.

## 2-3. 프로젝트 목표

- 한 번 입력한 제품·증상·조치 정보를 상담과 방문 단계까지 재사용한다.
- 공식 근거와 안전 규칙을 바탕으로 안내하고, 필요한 문의는 상담으로 전환한다.
- 상담·방문·후속 결과를 동일 문의 ID에 연결하여 전체 처리 흐름을 추적한다.

## 3-3. 핵심 차별점

> **핵심 차별점은 공식 근거 기반 안내와 선택적 상담 전환, 상담·방문·후속 결과를 동일 문의 ID로 연결하는 연속적인 업무 흐름이다.**

1. 한 번 입력한 제품·증상·조치 정보를 상담과 방문 단계까지 재사용한다.
2. 제품 모델과 문서 버전을 제한한 공식 문서 RAG와 명시적 안전 규칙을 적용한다.
3. 고객 답변을 역할별 업무 자료로 변환하고, 방문 결과와 해결 여부를 원 문의에 환류한다.

| 구분             | 대상                               | 제공 가치                                                        |

## 3-5. 기대 가치와 검증 방향

# 4. 시스템 구성 기획

## 4-1. 서비스 아키텍처 다이어그램

> **서비스 아키텍처 다이어그램의 텍스트 변환**

> ```text
> 고객 화면 ─┐
> 상담사 화면 ├─> 백엔드 API·업무 오케스트레이터
> 기사 화면 ──┤      ├─ 인증·권한 및 상태 관리
> 운영 화면 ──┘      ├─ Agent 실행 순서 제어
>                     ├─ 데이터 저장·조회
>                     └─ 화면별 결과 조립
>                              ↓
>                 증상 구조화·안전 규칙 계층
>                              ↓
>                 공식 문서 RAG·근거 검증 계층
>                              ↓
>                 역할별 결과 생성·검증 계층
>                              ↓
>          관계형 DB · Vector DB · 공식 근거 레지스트리
>                              ↓
>       고객 안내 · 상담 요약 · 기사 리포트 · 운영 현황
> ```

> 상담·방문 결과는 다시 원 문의와 케어 이력에 반영되어 후속 상태 확인과 다음 케어 일정 갱신에 사용된다.

## 4-2. 구성 요소

| 구성 요소                      | 주요 역할                                                                                     |
|--------------------------------|-----------------------------------------------------------------------------------------------|
| 역할별 화면                    | 고객, 상담사, 방문기사, 운영 담당자가 역할에 맞는 입력·조회·처리를 수행한다.                  |
| 백엔드 API·업무 오케스트레이터 | 인증·권한, 문의 상태 관리, Agent 실행 순서 제어, 데이터 저장·조회, 화면 결과 조립을 담당한다. |
| 증상 구조화·안전 규칙 계층     | 고객 입력을 구조화하고 위험 신호·사용 제한 여부를 우선 판단한다.                              |
| 공식 문서 RAG·근거 검증 계층   | 제품 모델·세대에 맞는 공식 문서를 검색하고, 근거 적용 가능성을 검증한다.                      |
| 역할별 결과 생성·검증 계층     | 고객 안내, 상담 요약, 기사 리포트를 생성하고 근거·안전·적용성을 다시 검토한다.                |
| 데이터 저장소                  | 관계형 DB, Vector DB, 공식 근거 레지스트리를 통해 문의 이력과 공식 근거를 관리한다.           |

## 4-3. 핵심 데이터 흐름

1. 고객 입력과 추가 답변을 동일 inquiry_id에 누적한다.
2. 증상을 구조화하고 안전 규칙을 먼저 적용한 뒤, 제품별 공식 문서를 검색·검증한다.
3. 검증된 근거로 고객 안내·상담 요약·기사 리포트를 생성하며, 상담·방문 결과는 원 문의에 환류한다.

## 4-4. 상태 관리

| 상태 구분      | 관리 대상                       | 예시 상태                                                                     |
|----------------|---------------------------------|-------------------------------------------------------------------------------|
| 문의 진행 상태 | 접수·상담·방문·완료의 업무 단계 | 작성 중, 문진 진행, AI 안내, 상담 대기, 상담 진행, 방문 예정, 완료 대기, 해결 |
| 방문 진행 상태 | 기사 배정과 방문 일정 세부 정보 | 기사 배정 중, 일정 조율 중, 방문 확정, 방문 진행, 방문 완료                   |

## 4-5. 구현 원칙

- 업무 상태는 백엔드 State Machine이 관리하고 변경 이력을 저장한다.
- 위험 신호와 근거 부족 시 자가조치보다 상담 전환을 우선한다.
- 공식 근거와 AI 생성 결과를 구분하고, 실패 시 기존 입력을 보존한다.
- 고객 입력부터 방문 결과까지 동일 inquiry_id로 연결한다.
