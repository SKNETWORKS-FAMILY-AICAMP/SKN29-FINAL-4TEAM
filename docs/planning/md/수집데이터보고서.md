## 2.2 수집 도구·자동화 및 예외 처리

| 항목 | 내용 |
| --- | --- |
| 사용 언어 / 라이브러리 | `Python 3` / `pypdf` + `csv` + `json` + `hashlib` |
| 자동화 여부 및 주기 | 공식 매뉴얼·FAQ를 새로 수집하거나 대상 모델과 선별 기준을 변경한 경우, `scripts/extract_selected_model_data.py`를 실행하여 구조화 데이터를 갱신한다. 이 스크립트는 수집된 공식 매뉴얼을 페이지별 텍스트로 추출하고, 대표 증상 관련 페이지와 FAQ 후보를 선별한 뒤 출처·버전·파일 해시 등의 메타데이터를 저장한다.<br><br>공식 PDF 다운로드와 핵심 페이지의 시각 검수는 수동으로 수행한다. |
| 오류 발생 시 예외 처리 전략 | 수집에 실패하면 판매 코드, 계열 코드, 변형 코드 순으로 다시 검색하고 실패 로그와 미확보 목록에 기록한다.<br><br>파싱 오류가 발생한 자료는 검증이 완료될 때까지 전처리 데이터와 RAG 대상에서 제외한다.<br><br>모델이 일치하지 않는 경우 판매 코드, 매뉴얼 코드와 제품 세대를 별도의 분류 필드로 관리하며 D형과 S형 문서를 혼용하지 않는다.<br><br>모델이 표기되지 않은 FAQ와 시연용 메타데이터는 공식 근거로 사용하지 않는다. |

## 2.3 수집·전처리·검수 흐름

1. **수동 수집**  
   공식 자료 다운로드 → 원본 폴더 저장 → 제품·문서 계보 확인
2. **자동 전처리**  
   페이지별 본문 추출 → 증상 키워드 검색 → FAQ 후보 및 범위 분류 → 구조화 데이터·메타데이터 생성 → 해시값 기록
3. **수동 검수·분리**  
   핵심 페이지 시각 검수 → 적용성 최종 확인 → MVP용·확장용·시연용 데이터 분리

# 5. 데이터 품질 및 정합성 관리 방안

| 항목 | 관리 방안 |
| --- | --- |
| 중복 제거 기준 | 원본 파일은 SHA-256 해시값을 비교하여 중복 여부를 판별한다. 가공 데이터는 유형별 고유 식별자를 사용한다. 매뉴얼 페이지는 문서 ID와 페이지 번호의 조합, RAG 청크는 청크 ID, FAQ는 FAQ ID, 합성 시나리오는 시나리오 ID, 근거 레지스트리는 근거 ID를 기준으로 중복을 확인한다. |
| 정합성 검증 방법 | 원본 검증: PDF의 파일명, 버전, 페이지 수, 파일 크기, 해시값과 공식 등록일을 확인한다.<br>형식 검증: JSON, JSONL과 CSV 파일이 오류 없이 읽히는지 확인한다.<br>참조 검증: 시나리오가 참조하는 RAG 청크가 실제로 존재하는지 확인하고, RAG의 문서 ID와 페이지 번호가 매뉴얼 페이지 데이터와 연결되는지 검사한다.<br>적용 범위 검증: 판매 코드와 매뉴얼 모델 코드, 문서 세대, 모델 적용성 및 사용 허용 범위가 서로 일치하는지 교차 확인한다. |
| Null 처리 및 결측치 전략 | 필수 식별자, 모델, 출처 또는 검증 상태가 누락된 데이터는 저장하지 않는다. 선택 항목이 없으면 빈 배열로 저장한다. 모델 세대나 개별 주소를 확인할 수 없는 FAQ는 임의로 추정하지 않고 ‘미검증’ 또는 ‘조건부 사용’으로 분류한다. |
| 표준화 전략 | 문자 인코딩은 UTF-8, 날짜는 YYYY-MM-DD, 경로 구분자는 슬래시(/)로 통일한다. 판매 코드는 공식 판매 페이지의 표기대로 하이픈 없이 저장하고, 매뉴얼 모델 코드는 공식 문서의 하이픈 포함 표기를 유지한다. 또한 활용 범위, 모델 적용성, 문서 세대, 검증 상태와 사용 허용 범위는 사전에 정의한 공통 코드값을 사용한다. |
