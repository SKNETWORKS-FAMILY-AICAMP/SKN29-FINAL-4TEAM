## 7월 21일(화) - 역할별 산출물 작성 및 PM 피드백 반영
* **진행 업무**
  * 역할별 담당 산출물 작성 및 팀 공유
  * PM의 산출물 검토와 요구사항 정합성 확인
  * 피드백에 따른 역할별 문서 수정 및 재공유
  * 프로젝트 대상 모델과 대표 증상 범위 통일
* **주요 작업 내용**
  * 각 팀원이 시장조사·BM 분석, 시스템·AI·RAG 구조, 데이터 수집 전략, 데이터 구조·품질 기준, UX 화면구성안 등 담당 산출물을 작성하여 공유함.
  * PM이 각 산출물을 요구사항 정의서와 비교하여 누락 기능, 사용자별 입력·조회 권한, 상태 코드, 화면 흐름 및 데이터 필드의 일치 여부를 검토하고 피드백함.
  * 문의 진행 상태, 방문 일정 상태, AI 처리 상태를 분리하고, `COMPLETION_PENDING`, `REVISIT_REQUIRED`, `REOPENED`를 추가하여 완료 대기·추가 방문·문의 재개를 구분함.
  * 방문 일정 상태는 기사 배정 중, 일정 조율 중, 방문 확정, 방문 진행 중, 방문 완료로 세분화함.
  * 고객은 자가조치 수행 여부와 결과를 입력하고 상담을 요청하는 단계까지만 수행하도록 수정함.
  * 방문 필요 여부 판단, 고객 방문 희망일, 가상 담당 기사, 일정 상태 및 가상 방문 확정일 등록은 상담사가 수행하고 고객은 등록된 일정을 조회하도록 역할을 구분함.
  * 위험 문의는 자가조치 안내를 차단하고 상담으로 우선 전환하되, 위험 신호와 공식 근거에 따라 `PARTIAL_STOP`, `TOTAL_STOP`, `PENDING_CONSULTATION`으로 사용 제한 범위      를 구분함.
  * 정수기 대상 모델은 주 MVP `WPU-JAC104D 계열`, 후속 확장 `WPU-IAC425 계열`로 통일하고, 기존 `WPU-IAC506` 관련 자료는 신규 구현에서 제외하도록 정리함.
  * 대표 증상은 출수량 저하, 물맛·냄새 이상, 누수, 냉·온수 온도 이상 네 가지로 통일함.
  * PM 피드백을 반영한 수정본을 다시 공유하고, 화면·데이터·시스템 문서 간 상태 코드, 역할 권한, 모델명 및 필드명을 교차 검토함.

   * 고객의 증상 입력, AI 안내, 상담 전환, 방문점검, 결과 확인으로 이어지는 전체 과정을 실제 서비스 화면 형태로 시각화함.

   * 각 사용자 화면에서 입력된 요청이 백엔드 API와 업무 오케스트레이터를 거쳐 처리되는 전체 시스템 아키텍처 구조도를 작성함.
   * AI 처리 흐름을 증상 구조화, 안전 규칙 판단, 공식 문서 RAG 검색, 근거 검증, 역할별 결과 생성, 최종 결과 검증 순서로 구성함.

   * 모든 요청과 처리 결과를 동일한 ID로 연결하여 고객이 한 번 제공한 정보가 상담 및 방문 단계에서도 지속적으로 활용되는 구조를 핵심 원칙으로 반영함.

## 7월 24일(금) - 기술스택·API 명세 정리 및 개발 실행계획 수립
* **진행 업무**
   * 팀 WBS와 공통 개발 규칙을 기준으로 프로젝트 기술스택 정의서 작성
   * 사용자 채널별 기술 검토안과 백엔드·AI·RAG·DB·배포 구조 교차 검토
   * WaterCare Public·Internal API 명세 작성 및 기술스택 기준 반영
   * 역할별 주담당·협업 업무계획과 우선순위, 완료 증거 및 결정 대기 사항 정리
   * 2주차 진행 내용과 핵심 산출물·프로토타입을 활용한 주간 발표자료 구성

* **주요 작업 내용**
   * 업무 백엔드는 Django·Django REST Framework, AI·RAG API는 FastAPI·Pydantic, 데이터베이스는 PostgreSQL을 중심으로 책임을 분리하고, 키워드 검색용 FTS·GIN과 벡터 검색용 pgvector를 함께 사용하는 하이브리드 검색 구조를 정리함.
   * 상담사·운영직원 Web은 React·TypeScript·Vite, 고객 모바일은 Android Native·Kotlin·Jetpack Compose 담당자안을 반영함. 방문기사 태블릿은 Android Native와 Web 중 팀 결정이 필요하므로 미정 상태로 유지함.
   * 인증·권한은 JWT·RBAC, 배포는 Docker image·Kubernetes·GitHub Actions, 추적은 구조화 JSON 로그와 `correlation_id`를 적용 기준으로 정리함.
   * 기술스택을 필수·개발 및 검증·조건부·제외 기술로 구분하고, JWT 수명주기, 날짜·시간의 시간대, 도메인형 문자열 ID 생성 규칙, Enum·Seed 방식, 백업 정책 등 팀 합의가 필요한 항목은 임의로 확정하지 않고 결정 대기로 분리함.
   * API 명세는 Public API 39개와 Internal API 5개, 총 44개의 Method·Path를 정리하고 REST `/api/v1`, 공통 응답 Wrapper, 오류 형식, 페이지 번호 방식, `allowed_actions`, `state_version`, `idempotency_key`, `EvidenceCardDTO` 등 공통 계약을 반영함.
   * API 계약 상태를 `ALIGNED` 21개, `CONDITIONAL` 7개, `OPEN` 12개, `BLOCKED` 4개로 구분함. 아직 구현·Migration·OpenAPI·자동 테스트·리뷰 증거가 없으므로 전체 구현 상태는 `NOT_IMPLEMENTED`, 문서 판정은 조건부 적합으로 기록함.
   * 고객 보유 제품과 선택 구독 저장 모델, 기사 사전 확인 저장 필드, 도메인형 문자열 ID와 기존 UUID 설계의 정합성 등 구현을 막는 항목을 식별하고 우선 결정 순서를 정리함.
   * 업무계획표에는 주담당 16개와 협업·검토 25개를 구분하고, 진행 중인 `T-005` 데이터 설계와 `T-016` 백엔드 공통 구조의 마감, `T-017` 사용자 인증·권한 착수를 우선 실행 대상으로 정리함.
   * 문서 작성이나 설계 보유만으로 완료 처리하지 않고, 구현·테스트·리뷰·결과 파일 또는 리포트가 모두 확인될 때 완료로 인정하도록 증거 기준을 명시함.
   * 기술스택 정의서와 API 명세서를 Markdown·Excel 산출물로 정리하고, 2주차 발표자료에는 핵심 사용자 인계 흐름, ERD, 역할별 프로토타입과 근거 자료를 연결함.
