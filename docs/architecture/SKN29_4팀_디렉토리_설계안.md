# SKN29 4팀 디렉터리 설계안

> 대상 프로젝트: **정수기 구독 고객 케어 및 A/S 업무 지원 시스템**  
> 설계 기준: **단일 Git 모노레포 + 최상위 기술 영역 분리 + 각 영역 내부 업무 기능 분리**  
> 확정 기술: 웹 `React + TypeScript + Vite + React Router`, 백엔드 `Python + Django + Django REST Framework`, 관계형 DB `PostgreSQL`  
> 미확정 기술: 모바일 프레임워크, AI 프레임워크·모델, Vector DB, 배포 방식

## 1. 확정 기술 스택 반영

백엔드는 다음 기술 스택으로 확정한다.

```text
백엔드: Python + Django + Django REST Framework
```

따라서 백엔드는 Django 프로젝트 설정과 업무 도메인별 Django App을 분리하고, 각 App 내부를 다음 책임으로 나눈다.

```text
DRF View·ViewSet    → HTTP 요청·응답과 권한 확인
Serializer          → 요청 형식 검증과 응답 직렬화
Service             → 상태 전이와 핵심 업무 규칙
Selector            → 조회 전용 ORM 쿼리
Model               → PostgreSQL 테이블과 관계 정의
```

Django View나 Serializer에 State Machine, 상담·방문 전환, AI 호출 같은 핵심 업무 로직을 직접 작성하지 않는다.

## 3. 핵심 설계 원칙

### 3.1 최상위는 역할별, 내부는 기능별

- 최상위는 `frontend/`, `backend/`, `ai/`, `data/`, `database/`처럼 담당 영역별로 구분한다.
- 각 영역 안에서는 `inquiry`, `consultation`, `visit`, `care`, `evidence`처럼 업무 기능별로 구분한다.
- 화면·API·DB·AI 파일이 한 폴더에 섞이지 않으면서도, 기능 단위 수정 범위를 찾기 쉽다.

### 3.2 공통 계약은 실행 코드가 아니라 `contracts/`에 둔다

프론트엔드는 TypeScript, 백엔드는 Python을 사용하고 AI 영역의 구현 방식은 아직 확정되지 않았으므로, 루트 `shared/`에 특정 언어의 실행 코드를 두지 않는다. 대신 다음을 언어 중립 형식으로 관리한다.

- OpenAPI
- JSON Schema
- 요청·응답 예시 JSON

프론트엔드는 TypeScript 타입으로, 백엔드는 DRF Serializer·Python 타입으로 해당 계약을 매핑한다.

### 3.3 계층 의존 방향

```text
frontend
  → backend API

backend api
  → backend services
  → backend repositories
  → PostgreSQL

backend services
  → ai integration client
  → AI orchestrator
  → retrieval / guards / generation / validation

contracts
  → frontend·backend·ai가 함께 참조하는 데이터 규격
```

다음 역방향 의존은 금지한다.

```text
Repository → Service 호출 금지
AI 모듈 → 프론트엔드 코드 참조 금지
프론트엔드 → PostgreSQL 직접 접근 금지
프론트엔드 → data/ 파일 직접 조회 금지
화면 → 근거 레지스트리 원본 직접 조회 금지
```

### 3.4 주석 작성 원칙

- 폴더 책임은 각 영역의 `README.md`에 기록한다.
- 코드는 이름만으로 이해되는 내용을 반복해서 주석으로 쓰지 않는다.
- 상태 전이, 위험 분기, 근거 제외처럼 **왜 그렇게 처리하는지**가 중요한 로직에만 주석을 남긴다.
- 임시 작업은 다음 형식을 사용한다.

```text
TODO(T-000, 담당자, YYYY-MM-DD): 처리할 내용
```

## 5. 주의 사항

- `data/archive/`는 생성하지 않는다. 폐기된 `WPU-IAC506` 관련 데이터는 신규 저장소·RAG·화면·시연 범위에서 사용하지 않는다.
- `data/raw/`의 공식 PDF·FAQ 원문은 Git에 올리지 않는다.
- 공식 근거의 내부 경로와 원문 전체를 프론트 응답에 포함하지 않는다.
- DRF View·ViewSet은 요청·응답과 권한 확인만 담당하고, 상태 전이·상담·방문·케어 업무 규칙은 각 App의 `services.py`에 둔다.
- 조회 전용 ORM 쿼리는 `selectors.py`에 두고, View나 Serializer에서 복잡한 QuerySet을 직접 조립하지 않는다.
- 상태 전이와 중복 요청 방지는 `transaction.atomic`, `state_version`, `idempotency_key`를 함께 사용하도록 설계한다.
- Django Migration은 모델 변경과 함께 커밋하며, 운영 DB를 수동으로 먼저 변경하지 않는다.
- 빈 폴더는 Git에서 추적되지 않으므로, 기술 미확정 영역은 `README.md` 또는 필요한 경우 `.gitkeep`만 둔다.
