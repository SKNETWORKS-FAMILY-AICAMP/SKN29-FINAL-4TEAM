## 구조도의 핵심 목적

이 화면설계서를 구조도로 옮길 때는 모든 화면·상태·필드를 넣기보다, 보는 사람이 다음 네 가지를 바로 이해하게 해야 합니다.

> **누가 사용하는가 → 정보가 어떤 순서로 이동하는가 → AI가 어디서 무엇을 처리하는가 → 공식 근거와 결과가 어디에 저장되는가**

현재 화면설계서는 고객·상담사·방문기사 흐름, AI 처리 상태, 안전 규칙, RAG 근거 레지스트리까지 상당히 세밀하게 정의되어 있습니다. 따라서 구조도에서는 이 내용을 전부 펼치기보다 **업무 흐름과 기술 구조를 한 장 안에서 계층적으로 축약**하는 것이 좋습니다.

# 추천 구조: 4단 아키텍처

## 1단: 사용자 영역

맨 위에는 사용자 네 종류만 배치합니다.

```text
고객        상담사        방문기사        운영 담당자
```

각 사용자 아래에는 대표 행동만 2~3개씩 표시합니다.

| 사용자    | 구조도에 표시할 핵심 행동                  |
| ------ | ------------------------------- |
| 고객     | 제품·케어 확인, 증상 입력, 조치 결과·해결 여부 입력 |
| 상담사    | 문의 요약 확인, 상담 기록, 방문 전환          |
| 방문기사   | 사전 점검 리포트 확인, 방문 결과 등록          |
| 운영 담당자 | 문의·오류·처리 현황 조회                  |

화면 ID인 `CUST-01`, `CONS-02`, `TECH-03` 등을 전부 표시하면 오히려 이해가 어려워집니다. 구조도에서는 **고객 화면**, **상담사 화면**, **기사 화면** 정도로만 표시하고, 화면 ID는 하단 주석이나 별도의 화면 흐름도에서 다루는 것이 좋습니다.

---

## 2단: 서비스 업무 흐름

구조도의 중앙에는 프로젝트의 핵심 차별점인 **정보 인계 흐름**을 가장 크게 그립니다.

```text
제품·케어 정보
      ↓
고객 증상·문진
      ↓
AI 분석·공식 근거 안내
      ↓
상담사용 문의 요약
      ↓
방문기사용 사전 점검 리포트
      ↓
방문 결과·케어 이력 반영
      ↓
고객 해결 여부 확인
```

이 흐름에서 반드시 강조할 것은 **모든 정보가 하나의 문의 ID로 연결된다**는 점입니다.

중앙을 관통하는 선이나 배지로 다음을 넣는 것이 좋습니다.

```text
동일 inquiry_id로 전체 이력 연결
```

이 한 줄이 프로젝트의 차별점을 가장 잘 보여줍니다.

### 위험 문의는 별도 빠른 경로로 표시

일반 문의와 위험 문의가 똑같이 흘러가는 것처럼 보이면 안 됩니다.

```text
일반·주의
→ 공식 근거 기반 자가 확인
→ 조치 결과 입력

위험·근거 부족
→ 안전 안내
→ 즉시 상담 전환
```

위험 경로는 굵은 화살표나 점선 분기로 표시합니다.

```text
위험 감지
 └────────→ 안전 안내 → 상담사
```

## 3단: 시스템 처리 영역

서비스 흐름 아래에 실제 시스템 구성 요소를 배치합니다.

권장 구성은 다음 여섯 개입니다.

```text
역할별 웹 화면
        ↓
백엔드 API·업무 오케스트레이터
        ↓
┌─────────────────────────────┐
│ 증상 구조화·추가 질문       │
│ 안전 규칙·사용 안내 판단    │
│ 공식 문서 RAG 검색          │
│ 고객 안내 생성              │
│ 상담 요약·기사 리포트 생성  │
│ 결과 검증                   │
└─────────────────────────────┘
```

### 시스템 처리 영역에서 반드시 보여줄 모듈

#### ① 백엔드 API·업무 오케스트레이터

이 모듈은 전체 흐름의 중심입니다.

표시할 역할은 다음 정도면 충분합니다.

```text
- 사용자 권한 확인
- 문의 상태 관리
- Agent 실행 순서 제어
- 데이터 저장·조회
- 화면용 결과 조립
```

#### ② 증상 구조화·추가 질문

```text
고객 자연어
→ 증상 유형
→ 발생 조건
→ 대상 출수
→ 동반 증상
→ 누락 정보
```

#### ③ 안전 규칙 엔진

이 부분은 LLM과 반드시 분리해서 표현하는 것이 좋습니다.

```text
안전 규칙 엔진
- 위험 신호 우선 판정
- 직접 분해·수리 안내 차단
- 현재 사용 안내 상태 결정
- 위험 시 상담 강제 전환
```

현재 사용 안내 상태 네 가지는 구조도에서 작은 배지로 표현하면 좋습니다.

```text
일반 사용 가능
일부 기능 사용 중지
제품 전체 사용 중지
판단 보류·상담 필요
```

#### ④ RAG 검색

```text
제품 모델 + 증상
        ↓
공식 문서 검색
        ↓
관련 청크·페이지 반환
```

여기서 `제품 모델 필터`를 반드시 표시해야 합니다.

```text
WPUJAC104DWH 전용 검색 범위
```

D 세대와 S 세대 문서가 섞이지 않도록 설계했다는 것이 이 프로젝트의 중요한 신뢰성 요소이기 때문입니다.

#### ⑤ 응답·리포트 생성

하나의 LLM 영역으로 묶되 출력은 역할별로 나눕니다.

```text
고객용
- 쉬운 안전 안내
- 공식 근거 요약
- 다음 행동

상담사용
- 제품·증상·조치 요약
- 미확인 정보
- 공식 근거

방문기사용
- 방문 사유
- 우선 점검 항목
- 준비·안전 유의사항
```

#### ⑥ 결과 검증

```text
- 공식 근거 존재 여부
- 위험 문구 검사
- 금지 표현 검사
- 모델 적용성 검사
```

이 검증 모듈을 마지막에 배치하면 “AI가 임의로 답하는 시스템이 아니다”라는 점이 한눈에 드러납니다.

## 4단: 데이터·근거 저장 영역

맨 아래에는 저장소를 세 종류로 분리해서 그리는 것이 좋습니다.

```text
관계형 DB          Vector DB          공식 근거 레지스트리
```

### 관계형 DB

```text
고객·구독·제품
문의·문진·증상
고객 조치
상담 기록
방문 일정·방문 결과
케어 이력
상태 변경 이력
```

### Vector DB

```text
공식 매뉴얼 청크
FAQ 청크
증상·제품 메타데이터
페이지 정보
```

### 공식 근거 레지스트리

```text
문서 ID
근거 ID
제품 모델·세대
페이지
검증 상태
사용 허용 정책
공식 출처 URL
```

공식 근거 레지스트리는 Vector DB와 분리해서 그리는 편이 좋습니다.

* Vector DB: 관련 문서를 **검색**
* 근거 레지스트리: 검색된 근거가 화면에 사용 가능한지 **검증·보강**

```text
Vector DB 검색 결과
        ↓
공식 근거 레지스트리 검증
        ↓
EvidenceCardDTO
        ↓
고객·상담사·기사 화면
```

# 핵심 업무 흐름은 구조도 중앙에 별도로 강조

아키텍처 박스만 나열하면 일반적인 AI 서비스처럼 보일 수 있습니다. 따라서 중앙 또는 오른쪽에 핵심 업무 흐름을 세로로 배치하는 것이 좋습니다.

```text
고객
증상·문진 입력
   ↓
AI
구조화·위험 판단·공식 근거 검색
   ↓
상담사
문의 요약·방문 여부 결정
   ↓
방문기사
사전 리포트 확인·방문 결과 등록
   ↓
고객
처리 결과 확인·해결 여부 입력
```

각 단계 옆에 다음 문장을 넣으면 차별점이 더 명확해집니다.

```text
한 번 입력한 정보를 다음 담당자가 이어서 활용
```

# 구조도에 넣지 않는 것이 좋은 내용

다음 요소들은 중요하지만 한눈에 보는 아키텍처에는 넣지 않는 편이 좋습니다.

* 모든 화면 ID
* AI 처리 상태 8개 전체
* 문의 상태 코드 전체
* `EvidenceCardDTO`의 모든 필드
* 모든 시나리오 ID
* PDF 직접 링크 실패 절차
* `state_version`, `idempotency_key`
* 세부 오류 코드
* 각 매뉴얼 페이지 번호
* FAQ ID 전체

이런 내용은 별도의 표나 세부 흐름도에 두고, 메인 구조도에는 다음처럼 축약합니다.

```text
상태·이력 관리
오류·재시도 처리
공식 근거 검증
```

# 구조도에서 가장 강조해야 할 세 문장

시각적으로 강조할 문구는 세 개 정도면 충분합니다.

### 1. 정보 연결

> **고객이 한 번 입력한 정보를 상담과 방문까지 연결**

### 2. 공식 근거

> **제품별 공식 매뉴얼과 검증된 FAQ에 근거한 안내**

### 3. 안전 우선

> **위험 신호와 근거 부족 시 자가조치보다 상담 연결 우선**

이 세 문장이 구조도만 보더라도 읽히면, 다른 수강생이나 평가자가 프로젝트의 핵심을 바로 파악할 수 있습니다.
